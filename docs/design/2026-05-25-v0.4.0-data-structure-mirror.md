# v0.4.0 ‚Äî Full Data Structure Mirror (shioaji ‚Üî binance) ‚Äî Design

**Date**: 2026-05-25
**Status**: Draft (pending design review)
**Target release**: `binance-shioaji-sdk` v0.4.0 (breaking)
**Predecessor**: v0.3.0 (class rename `BinanceClient ‚Üí Binance`, complete method name mirror)
**Driver**: lcz-quant-core PR2 design surfaced that "mirror shioaji shape" was only **method-name deep** ‚Äî return types diverge (dict vs Pydantic object; combined vs decomposed). Adapter code for downstream consumers becomes asymmetric.

---

## 1. Context & motivation

### What was claimed (v0.3.0 release notes / README)

> "Async Python SDK for Binance USDM-M Futures, **mirroring the shioaji SDK API shape**. If you already trade Âè∞ËÇ°ÊúüË≤® via shioaji, this gives you Binance crypto with the **exact same API surface** ‚Äî same Contracts.Perp, same Order(action=‚Ä¶), same place_order. Switch contracts, keep your strategy code."

### What's actually mirrored vs not

| Aspect | Mirrored? | Evidence |
|---|---|---|
| Class name (`Binance` ‚Üî `Shioaji`) | ‚úÖ since v0.3.0 | `from binance_shioaji_sdk import Binance` |
| Method names (`place_order / cancel_order / login / account_balance / list_positions / list_trades`) | ‚úÖ | direct alignment with `sj.*` |
| Method arg signatures | ‚úÖ mostly | `place_order(contract, order)` etc |
| `Contracts.Perp` namespace (vs shioaji's `Contracts.Futures`) | ‚úÖ analogous | accessor pattern matches |
| **Return value types** | ‚ùå NOT mirrored | shioaji returns Pydantic objects; binance-shioaji-sdk returns dicts |
| **Field decomposition** (which model holds what) | ‚ùå NOT mirrored | shioaji splits `AccountBalance` + `Margin`; binance-shioaji-sdk merges into single dict |
| **Field names** | ‚ùå NOT mirrored | shioaji `code/price/pnl`; binance-shioaji-sdk `symbol/avg_price/unrealized_pnl` |
| **Status enum values** | ‚ùå NOT mirrored | shioaji `Submitted/PartFilled/Filled/Cancelled/Failed`; binance-shioaji-sdk `submitted/partially_filled/filled/cancelled/rejected` |

**Verdict**: mirror is shallow. Consumers (lcz-quant-core BrokerAdapter layer being the first real one) cannot write symmetric code over both vendors; each adapter must know vendor-specific field shapes.

### Why fix it now

lcz-quant-core PR2 brainstorm hit this directly when writing `ShioajiAdapter` + `BinanceAdapter`. Three rounds of plan-review caught real bugs (wrong dict keys; `account_balance()` doesn't have `available_margin` on shioaji side; etc.). Symmetric adapter code is impossible until SDK returns mirror-aligned types.

The "complete" mirror lets adapter code look like:
```python
async def get_balance(self) -> Balance:
    ab = await maybe_await(self._session.api.account_balance)         # both: AccountBalance object
    mg = await maybe_await(self._session.api.margin, account)         # both: Margin object
    return Balance(
        currency=...,
        equity=Decimal(str(ab.acc_balance)),
        available_margin=Decimal(str(mg.available_margin)),
        initial_margin=Decimal(str(mg.initial_margin)),
        maintenance_margin=Decimal(str(mg.maintenance_margin)),
    )
```
Identical body across `ShioajiAdapter` and `BinanceAdapter` ‚Äî only the `maybe_await` glue + `currency` constant differ.

## 2. Scope

### In scope (v0.4.0, breaking)

1. **New dataclasses** mirroring shioaji return models:
   - `BinanceAccountBalance` ‚Äî mirror of `sj.AccountBalance` (essential fields only)
   - `BinanceMargin` ‚Äî mirror of `sj.Margin` (essential margin/equity fields; not all 25 obscure ones)
   - `BinanceFuturePosition` ‚Äî mirror of `sj.FuturePosition`
   - `BinanceTrade` ‚Äî mirror of `sj.Trade` (composite: contract + order + status)
   - `BinanceTradeStatus` ‚Äî mirror of `sj.Trade.status` substructure
   - `BinanceOrderStatusEnum` ‚Äî mirror of `sj.OrderStatus` enum (CamelCase values: `Submitted / PartFilled / Filled / Cancelled / Failed`)
2. **Return-type changes** (breaking):
   - `account_balance()` ‚Üí `BinanceAccountBalance` (was `dict`)
   - `list_positions(account)` ‚Üí `list[BinanceFuturePosition]` (was `list[dict]`)
   - `place_order(contract, order)` ‚Üí `BinanceTrade` (was `OrderResponse`)
3. **NEW method**: `margin(account)` ‚Üí `BinanceMargin` (currently missing; required to mirror shioaji's `sj.margin(account)`)
4. **Field-name alignment** (within new dataclasses): use shioaji vocabulary ‚Äî `code` not `symbol`, `price` not `avg_price`, `pnl` not `unrealized_pnl`, `acc_balance` not `equity` in AccountBalance
5. **Enum alignment**: `BinanceOrderStatusEnum.Submitted` (CamelCase to match `sj.OrderStatus.Submitted`), not `"submitted"` lowercase string
6. **Existing `OrderResponse` deprecated** ‚Äî emits `DeprecationWarning` for one release, removed v0.5.0. Callers should migrate to `BinanceTrade`.
6a. **Binance-only data structure mimicry** (¬ß3.7) ‚Äî `funding_rate / funding_rate_history / open_interest` return typed dataclasses (not dicts); `ExecutionReport` renamed `BinanceFillReport` (alias kept one release); `BinanceMarketDataError` raised on REST failure (no `{error, ...}` dict returns).
7. **README + examples** updated to reflect new return types
8. **Version bump** 0.3.0 ‚Üí 0.4.0; tag + GitHub release

### Out of scope

- shioaji Margin has 25+ fields (collateral_amount, deposit_withdrawal, fee, future_open_position, future_settle_profitloss, option_*, royalty_revenue_expenditure, security_collateral_amount, today_*, yesterday_*, etc.). Mirror **essential subset only** (~6 fields). Binance API doesn't have direct equivalents for most; we don't fake them.
- shioaji Stock-specific models (`StockPosition`, `StockProfitLoss`, etc.) ‚Äî Binance is futures-only, no stock equivalent needed.
- shioaji `sj.Order` field-by-field mirror ‚Äî current binance Order dataclass (`price/quantity/action/price_type/...`) is fine; rename is optional.
- WebSocket / Quote subsystem changes ‚Äî out of scope (v0.4.0 is REST data-shape only).
- Backwards-compat shim that returns dict on old method while object on new ‚Äî too messy; clean break with one deprecation cycle.

### 3.4 Trade + TradeStatus + Order (place_order return)

`sj.Trade` is a composite: `trade.contract`, `trade.order`, `trade.status`.

`BinanceTrade` mirrors this structure:

```python
@dataclass(frozen=True)
class BinanceTradeStatus:
    """Mirror of sj.OrderStatusInfo ‚Äî order lifecycle snapshot.

    Broker-assigned order id lives HERE (`status.id`) per shioaji
    decomposition ‚Äî NOT on Order. Consumers access via `trade.status.id`.
    """
    id: str                              # broker order id (mirrors sj.OrderStatusInfo.id)
    status: BinanceOrderStatusEnum       # Submitted / PartFilled / Filled / Cancelled / Failed
    status_code: str                     # broker reason code (Binance: empty on success)
    order_datetime: str                  # ISO timestamp (Binance: updateTime epoch ms ‚Üí ISO)
    deal_quantity: Optional[int] = None  # cumulative filled qty (Binance: executedQty); Optional matches sj.OrderStatusInfo.deal_quantity type
    order_quantity: Optional[int] = None # original submitted qty (Binance: origQty)
    cancel_quantity: Optional[int] = None # qty remaining after cancellation (orig - executed when cancelled)
    modified_price: float = 0.0          # weighted avg fill price (0.0 if no fill)
    msg: str = ""                        # broker message (error reason on failure)

@dataclass(frozen=True)
class BinanceTrade:
    """Mirror of sj.Trade. Returned by place_order().

    Composite: contract + order (user input) + status (broker snapshot).
    Consumer accesses order id as trade.status.id (NOT trade.order.id) ‚Äî
    matches shioaji idiom exactly.
    """
    contract: BinanceContract            # the contract that was traded (existing class)
    order: Order                         # the order params submitted (user input; no id field per shioaji)
    status: BinanceTradeStatus           # current status snapshot (carries broker order id)
```

Fields rationale:

| sj.OrderStatusInfo field | BinanceTradeStatus field | Binance source |
|---|---|---|
| `id` | `id` | `orderId` from POST `/fapi/v1/order` |
| `status` (OrderStatus enum) | `status` (BinanceOrderStatusEnum) | `status` field of order response (mapped per ¬ß3.5) |
| `status_code` | `status_code` (default `""`) | not directly in Binance response; populated on error path |
| `order_datetime` | `order_datetime` | `updateTime` epoch ms ‚Üí ISO 8601 |
| `deal_quantity` | `deal_quantity` | `executedQty` |
| `order_quantity` | `order_quantity` | `origQty` |
| `cancel_quantity` | `cancel_quantity` | computed: `orig - executed` when status in {Cancelled, Failed} else 0 |
| `modified_price` | `modified_price` | `avgPrice` (0.0 if no fill ‚Äî but if `status=Filled` and `avgPrice=0.0`, treat as broker data corruption per consumer responsibility) |
| `msg` | `msg` | empty on success; error message body on Binance HTTP 4xx |
| (`deals`, `modified_time`, `web_id` omitted) | n/a | `deals` would need myTrades roundtrip; `modified_time` ‚âà `order_datetime`; `web_id` is shioaji-internal |

**Order id placement (corrected rationale)**:

shioaji's broker-assigned order id lives on `Trade.status.id` (i.e., `sj.OrderStatusInfo.id`) ‚Äî verified live `dir(sj.OrderStatusInfo)` contains `id`. v0.4.0 mirrors this: `BinanceTradeStatus.id` carries the Binance `orderId`. Consumer access pattern `trade.status.id` matches shioaji.

**Deliberate divergence ‚Äî `BinanceTrade.order: Order` (user-input only)**:

shioaji actually stores `Trade.order` as `sj.OrderResult` (broker-enriched), NOT `sj.Order` (user-input). `sj.OrderResult` has extra fields **`id / seqno / ordno / ca`** that `sj.Order` does not ‚Äî these are shioaji-internal broker tracking numbers populated after submission.

We deliberately do NOT create a `BinanceOrderResult` wrapper, because:
- **Binance has no equivalent of `seqno` / `ordno`** (shioaji broker-internal sequence numbers). Binance's only post-submit identifier is `orderId` ‚Äî already captured in `BinanceTradeStatus.id`.
- Creating an empty `BinanceOrderResult` with only `id` populated (other fields None) would be **fake mirror** ‚Äî pretending to have shioaji parity without real data.
- `Order` as user-input struct is the honest shape: caller submits via `Order(...)`, gets back the original `Order` in `BinanceTrade.order` (unchanged), reads enrichment via `BinanceTradeStatus`.

Consumers needing shioaji-side `seqno` / `ordno` access must `isinstance(trade.order, sj.OrderResult)` branch ‚Äî no symmetric pattern possible because the data simply doesn't exist on Binance side. Per design philosophy: mirror what's mirrorable, honest divergence on what isn't.

### 3.5 BinanceOrderStatusEnum

Mirror `sj.OrderStatus` exactly (CamelCase, identical member names):

```python
class BinanceOrderStatusEnum(StrEnum):
    Cancelled = "Cancelled"
    Failed = "Failed"
    Filled = "Filled"
    Inactive = "Inactive"
    PartFilled = "PartFilled"
    PendingSubmit = "PendingSubmit"
    PreSubmitted = "PreSubmitted"
    Submitted = "Submitted"
```

Binance's `status` field returns: `NEW / PARTIALLY_FILLED / FILLED / CANCELED / REJECTED / EXPIRED`. Mapping:

| Binance API status | BinanceOrderStatusEnum |
|---|---|
| `NEW` | `Submitted` |
| `PARTIALLY_FILLED` | `PartFilled` |
| `FILLED` | `Filled` |
| `CANCELED` | `Cancelled` |
| `REJECTED` | `Failed` |
| `EXPIRED` | `Cancelled` (or new `Expired` if we want more granularity; for v0.4.0 collapse into Cancelled) |
| (no equivalent) | `Inactive`, `PendingSubmit`, `PreSubmitted` ‚Äî present in enum for shioaji parity but never produced by Binance |

### 3.6 list_trades return

`sj.list_trades()` returns `list[Trade]` (no separate "filled-trade history" model in shioaji ‚Äî same `Trade` shape is reused for both live and history). `binance.list_trades(...)` v0.3.0 returns broker-specific dicts. v0.4.0: **`list[BinanceTrade]`**.

**`BinanceTrade.contract` for history items** (semantic note): Binance `/fapi/v1/allOrders` returns a `symbol` string per order, not a full contract object. For history items, `BinanceTrade.contract` is a synthetic stub: `BinanceContract(symbol=entry["symbol"])` with all non-symbol fields at defaults. Full contract metadata (multiplier, exchange, etc.) requires a follow-up `Contracts.Perp[symbol]` lookup if needed. This is a **documented semantic difference** from `place_order` return (where `contract` is the fully-specified object the caller passed in). Adapter consumers reading `trade.contract.code` works for both paths; reading `trade.contract.multiplier` works only for live-submit results.

**Empty symbol argument**: `list_trades(symbol=None)` raises `ValueError("symbol required for Binance history query")` rather than silently returning `[]`. Diverges from shioaji's no-symbol return-all (Binance API requires `symbol` parameter). Document in method docstring + v0.4.0 release notes.

### 3.7 Binance-only features ‚Äî shioaji coding style mimicry

**Design philosophy (locked)**:
- **shioaji-only features ‚Üí don't force mirror** (Binance has no equivalent ‚Üí don't fake; already handled in ¬ß3.4 `seqno/ordno` divergence)
- **Binance-only features ‚Üí keep them, but mimic shioaji's coding style** (Pydantic-like attr-access object; shioaji vocabulary in field names; frozen dataclass not dict)

Binance has these features shioaji doesn't (crypto-specific concepts):
- `market_info.funding_rate(symbol) -> dict`            ‚Üê currently dict; **convert to dataclass**
- `market_info.funding_rate_history(...) -> list[dict]` ‚Üê currently list[dict]; **convert to list[dataclass]**
- `market_info.open_interest(symbol) -> dict`           ‚Üê currently dict; **convert to dataclass**
- `quote.subscribe_user_stream(...)` (no return shape change needed)
- `quote.wait_fill(...) -> ExecutionReport | None`      ‚Üê already typed; **rename `ExecutionReport` ‚Üí `BinanceFillReport` for naming consistency**
- `quote.subscribe(contract, channel, callback)` with channels `tick / book_ticker / kline_<iv>` (no return shape change ‚Äî callback-driven; out of v0.4.0 dataclass scope)

**New dataclasses (v0.4.0)** ‚Äî shioaji coding style (frozen dataclass, snake_case attrs, short field names like shioaji's `code/price/pnl` not verbose Binance API names):

```python
@dataclass(frozen=True)
class BinanceFundingRate:
    """Binance-only ‚Äî current funding rate snapshot.
    
    No shioaji equivalent (TW futures don't have funding). Field naming
    follows shioaji convention: `code` not `symbol`; `rate` not
    `lastFundingRate`; ISO datetime str not epoch ms.
    """
    code: str                   # e.g. "BTCUSDT" (shioaji vocab `code`, not `symbol`)
    rate: float                 # current period funding rate (e.g. 0.0001 = 0.01%)
    next_funding_time: str      # ISO 8601 UTC

@dataclass(frozen=True)
class BinanceFundingRateEntry:
    """One historical funding rate entry ‚Äî returned in list."""
    code: str
    rate: float
    funding_time: str           # ISO 8601 UTC (when this rate applied)
    mark_price: float           # mark price at funding (Binance returns; useful for analysis)

@dataclass(frozen=True)
class BinanceOpenInterest:
    """Binance-only ‚Äî open interest snapshot."""
    code: str
    open_interest: float        # contracts (BTC for BTCUSDT etc)
    open_interest_usdt: float   # USDT notional value
    timestamp: str              # ISO 8601 UTC
```

**Method signature updates**:

| Method | Before (v0.3.x) | After (v0.4.0) |
|---|---|---|
| `funding_rate(symbol)` | `‚Üí dict` | `‚Üí BinanceFundingRate` |
| `funding_rate_history(...)` | `‚Üí list[dict]` | `‚Üí list[BinanceFundingRateEntry]` |
| `open_interest(symbol)` | `‚Üí dict` | `‚Üí BinanceOpenInterest` |
| `wait_fill(order_id)` | `‚Üí ExecutionReport \| None` | `‚Üí BinanceFillReport \| None` (rename only) |

**Rename `ExecutionReport` ‚Üí `BinanceFillReport`** (class name only; field vocabulary explicitly exempt):
- Class name: `ExecutionReport` is generic FIX-protocol terminology; doesn't fit the rest of the SDK's naming convention (`BinanceContract`, `BinanceAccount`, `BinanceTrade`, etc.). Rename for consistency.
- shioaji equivalent for fills is `sj.Deal` (`price / quantity / seq / datetime / ts`) ‚Äî entirely different semantics from a Binance WebSocket execution event. No direct field-by-field mapping possible.
- **Field vocabulary exemption (H-1 documented)**: `BinanceFillReport` fields retain Binance WebSocket event vocabulary (`order_id`, `symbol`, `status`, `side`, `order_type`, `qty`, `filled_qty`, `last_filled_price`, `avg_price`). The shioaji-style coding principle applies to **class structure** (frozen dataclass, attr-access, typed) but NOT to field renaming for this class, because:
  1. `sj.Deal` has no `order_id` / `status` / `side` / `order_type` / `filled_qty` analog ‚Äî there's nothing to mirror
  2. Forcing renames (`symbol ‚Üí code`, `qty ‚Üí quantity`) without shioaji counterpart would be cosmetic-only and would obscure the WebSocket event's raw shape that downstream debugging relies on
  3. Two-axis principle applies coding-style mimicry where there's a model to style-match; `BinanceFillReport` has no model to style-match against
- **Deprecation mechanism**: `ExecutionReport = BinanceFillReport` in `__init__.py` as module-level alias; `__init__.py` adds `__getattr__` hook that emits `DeprecationWarning` on `from binance_shioaji_sdk import ExecutionReport` access. Removed v0.5.0. (Same pattern as `OrderResponse ‚Üí BinanceTrade`.)

**Unified error policy ‚Äî typed exceptions across the SDK (H-2 fix)**

Current SDK silently returns `{error, detail}` dict / zero-filled object / `[]` on various REST failures ‚Äî different policies per method, all lossy. v0.4.0 unifies under typed exception hierarchy:

```python
# binance_shioaji_sdk/exceptions.py (new file)
class BinanceSDKError(Exception):
    """Base for all v0.4.0+ typed errors."""

class BinanceMarketDataError(BinanceSDKError):
    """Raised when market_info endpoint returns API error or parse fail."""

class BinanceAccountError(BinanceSDKError):
    """Raised when account / position / balance endpoint returns API error."""

# Existing BinanceAuthError stays as-is (login failure)
```

**Error-axis applied across all v0.4.0 methods**:

| Method | v0.3.x error behavior | v0.4.0 error behavior |
|---|---|---|
| `account_balance()` | return `{equity: 0.0, ...}` zero-filled dict | raise `BinanceAccountError(detail)` |
| `margin(account)` (NEW) | n/a | raise `BinanceAccountError(detail)` |
| `list_positions(account)` | return `[]` on error | raise `BinanceAccountError(detail)` |
| `place_order(...)` | return `OrderResponse` with error in `raw` | raise `BinanceAccountError(detail)` (or auth subclass if 401) |
| `funding_rate(symbol)` | return `{error, detail}` dict | raise `BinanceMarketDataError(detail)` |
| `funding_rate_history(...)` | **return `[]` silently** | raise `BinanceMarketDataError(detail)` ‚Äî **behavioral break**, see ¬ß5 migration |
| `open_interest(symbol)` | return `{error, detail}` dict | raise `BinanceMarketDataError(detail)` |
| `wait_fill(order_id)` | return `None` on timeout | unchanged (timeout is NOT an error; `None` is sentinel) |

Principle: any REST endpoint **failure** raises; any REST endpoint **success-with-nothing** (e.g. genuinely zero positions, or genuinely 0 funding rate) returns the typed object with zeros. Distinct concepts.

**Why this matters**: a consumer reading `balance.equity == 0` should mean "account is empty", NOT "REST call failed silently". v0.3.x conflates the two; v0.4.0 separates via exception path.

**Acceptance addition to ¬ß6**:
- [ ] `funding_rate / funding_rate_history / open_interest` return typed dataclasses (no dict)
- [ ] `ExecutionReport` aliased to `BinanceFillReport` with `DeprecationWarning`
- [ ] `BinanceMarketDataError` raised on REST failure (no `{error, ...}` dict returns)
- [ ] New mirror test: assert Binance-only dataclasses follow shioaji naming convention (`code` not `symbol`; no verbose Binance API key names like `lastFundingRate` / `markPrice`)

## 4. Risks

| Risk | Mitigation |
|---|---|
| Binance allows fractional quantity (`0.001 BTC`); shioaji is integer contracts. `BinanceFuturePosition.quantity: int` truncates fractional positions. | Use `Decimal` instead of `int` for quantity in `BinanceFuturePosition`. Diverges from shioaji's int contract ‚Äî but trying to mirror "int" would silently round 0.5 BTC to 0. Honesty > mirror precision here. Document divergence in field docstring. |
| Binance has no native "position ID" ‚Äî synthesis `f"{symbol}_{positionSide}"` is fragile across hedge mode. | Default to one-way mode initially; document in v0.4.0 README. Hedge mode ‚Üí separate ID scheme in v0.5.0. |
| shioaji `Margin.today_balance` ‚âà `acc_balance`; redundant duplication in two models. | Keep for shioaji parity ‚Äî caller may query either; cheaper than re-fetching. |
| `BinanceOrderStatusEnum` has members never produced by Binance (`Inactive / PendingSubmit / PreSubmitted`). | Document as "shioaji-parity only" in docstring. Type system allows them; runtime never returns them from Binance side. |
| `BinanceTrade` composite breaks downstream callers of `OrderResponse`. | Deprecation cycle: v0.4.0 `place_order()` returns `BinanceTrade`; old `OrderResponse` accessible via `trade._legacy_response` property with `DeprecationWarning`. Removed v0.5.0. (Alternative: hard break in v0.4.0 since lcz-quant-core is the only known consumer and we control both repos.) |
| lcz-quant-core PR2 in-flight will need full rewrite of BinanceAdapter to consume new types. | Expected; that's the whole point of this release. PR2 design + plan get re-baselined post-v0.4.0. |
| New `margin(account)` method needs a Binance REST endpoint that returns margin breakdown. | `/fapi/v2/account` returns `totalInitialMargin / totalMaintMargin / availableBalance / totalMarginBalance`. Already used internally ‚Äî just expose as method. |

## 5. Backwards compatibility

**Decision: clean break + 1-release deprecation for `OrderResponse` only.**

Rationale:
- Only known consumer is lcz-quant-core (PR2 in-flight; we control it; bumps pin in coordinated PR)
- sentinel uses `lcz_binance_sdk` (different package, not affected)
- Maintaining dict-OR-object dual return for `account_balance / list_positions` adds 100+ lines of compat shim for zero real benefit

Migration path (in v0.4.0 release notes):
```python
# Before (v0.3.x):
balance_dict = await api.account_balance()
print(balance_dict["equity"])

# After (v0.4.0):
balance = await api.account_balance()
print(balance.acc_balance)              # mirror shioaji
# (margin info moved to separate call:)
mg = await api.margin(api.futures_account)
print(mg.available_margin)
```

